### What this proves

1. **Codex OAuth auth path works end-to-end.** The probe round-trips
   through Codex's API and receives a typed business-rule response (429
   quota), not a 401 / 403 / network error. The OAuth token is valid;
   reasoner-runtime correctly surfaces the structured response.
2. **reasoner_runtime classification is correct.** The probe sets
   `status=DEGRADED`, `reachable=False`, `retryable=True`, and assigns
   `quota_status=QuotaStatus.limited` — exactly the contract a downstream
   orchestrator would expect for a "fail closed but try again later"
   condition.
3. **Latency 1421ms** indicates a real network round-trip; not a local
   short-circuit.

### What this means for M2.6 readiness

The orchestrator's `DefaultReasonerRuntimeGateway.check_health()` call
(per `orchestrator/src/orchestrator_adapters/p2_dry_run.py:235-242`)
would observe `reachable=False` from the same probe. The orchestrator's
production daily-cycle policy treats Phase 2 LLM infrastructure failure
as **hard-stop** (orchestrator/CLAUDE.md Gate Behaviour Matrix:
"Phase 2 任意 LLM 基础设施级失败 → 硬停"). So an M2.6 daily-cycle proof
attempted today against the current Codex Plus plan would halt before
Phase 2 even invokes any L4/L6/L7 reasoner call.

## Unblocker options (pick one for M2.6)

### Option 1: Wait for Codex quota reset (~5.6 days)

Zero code or config change. Re-run the probe after ~2026-05-05; expect
`reachable=True` + `quota_status=QuotaStatus.OK`. This is the simplest
path if the M2.6 proof can wait.

### Option 2: Switch to MiniMax provider

Set in `assembly/.env` (or shell):

```
P2_REASONER_PROVIDER=minimax
P2_REASONER_MODEL=<MiniMax model id>
MINIMAX_API_KEY=<key>
MINIMAX_API_BASE=<optional base url>
```

Drop `REASONER_RUNTIME_ENABLE_CODEX_OAUTH=1`. reasoner_runtime's
LiteLLM-backed generic probe handles MiniMax; cf. M2.0 audit
finding §4 (reasoner_runtime PARTIAL → graceful-degradation health
probe with `reachable=False` on auth fail).

The compose dagster-daemon image already passes `MINIMAX_API_KEY` /
`MINIMAX_API_BASE` through (cf. `compose/lite-local.yaml:62-63`,
verified at M2.1 preflight).

> **Probe-path nuance (added in review fold):** MiniMax does **NOT** use
> the structured probe path Codex does. The set
> `_STRUCTURED_HEALTH_PROBE_PROVIDERS = frozenset({"openai-codex", "claude-code"})`
> at `reasoner_runtime/health/checker.py:26` excludes MiniMax. With
> `P2_REASONER_PROVIDER=minimax`, the health probe falls through to the
> generic LiteLLM `litellm.completion()` 1-token heartbeat
> (`checker.py:68-105`). Functional outcome is the same — the orchestrator
> still gets a `HealthCheckReport` with `reachable=True/False` — but the
> error classification on a quota / auth failure is coarser: the structured
> probe's `quota_status=limited|exhausted` distinction depends on
> Codex/Claude-Code-specific error parsing, while LiteLLM raw exceptions
> drop into the generic `provider_quota_status_from_error` heuristic. Not
> a blocker for M2.6 (reachable / unreachable is the only flag the
> orchestrator's hard-stop gate consumes per
> `orchestrator/src/orchestrator/checks/asset_checks.py:62-98`), but
> worth noting if the operator wants Codex-equivalent quota visibility
> from MiniMax.

### Option 3: Provision a second Codex account / upgrade plan

Operationally identical to Option 1 (Codex OAuth path), just with a
different `~/.codex/auth.json`. No reasoner-runtime code change.

### Recommendation

**Option 1 (wait)** is the lowest-effort path for M2.6's one-shot proof.
If the user wants to run M2.6 sooner, **Option 2 (MiniMax)** is the
fastest unblocker — provision a MiniMax key, set the two env vars, and
re-run the probe; the wiring chain is otherwise unchanged.
